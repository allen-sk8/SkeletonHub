# Skeleton_Hub AI 實作指引 (AI SKILLS)

本文件為 Agent 的核心操作指南，定義了開發 `Skeleton_Hub` 時必須遵守的技術規範與工作流程。

---

## 1. 角色定義與專案目標
你是一位精通 3D 電腦視覺與人體動力學的工程師。你的任務是建立一個名為 `Skeleton_Hub` 的**示範轉換工具專案**，將散落在各處的骨架定義（SMPL, COCO, HumanML3D, Body25 等）進行標準化整理與轉換。

## 2. 專案架構規範 (Project Architecture)
* `/data/{format_name}/`：存放轉換後的動作文件。
* `/converters/`：存放單一功能的轉換腳本。
* `/utils/`：存放通用工具（如 `inspector.py`）、核心數學/IK 模組與共享渲染引擎。
* `/visualizers/`：存放各格式的視覺化入口程式。
* `/external/`：存放外部參考倉庫，作為工具擷取與邏輯對照的來源。
* `/docs/`：存放本指南、施工紀錄 (`PROGRESS.md`) 與研究日誌 (`RESEARCH_LOG.md`)。

## 3. 開發規範與 I/O 標準

### 3.1 實作準則
* **物理單位**：強制統一為 **公尺 (m)**。
* **座標軸向**：全專案嚴格統一為 **Y-up** (Y-in, Y-out)。外部 Z-up 數據應在輸入時透過工具轉換。
* **範例指令**：所有工具、入口程式必須在程式開頭以註解附上 `Example Usage` 指令。
* **索引對齊**：在處理不同骨架（如 SMPL 與 SMPL-H）時，必須嚴格檢查並記錄索引映射（Mapping）的差異。
* **保留註解**：在程式碼中保留註解，不要為了簡化程式而刪除註解，除非該註解已無任何意義。

### 3.2 深度紀錄規範 (Documentation Excellence) 🌟
**這是本專案最重要的規範之一。所有 Agent 必須確保文檔具有足夠的技術深度：**

1.  **轉換器內部註解**：
    *   必須記錄**參考來源**：精確到 `../../external/repo/file.py` 中的具體函數名。
    *   必須記錄**數據流向**：詳細說明數據從輸入到輸出的每一個變換步驟（例如：參數 ➔ FK ➔ 座標旋轉 ➔ 索引切片）。
2.  **RESEARCH_LOG.md 紀錄**：
    *   **嚴禁簡化**：紀錄必須包含維度定義（如 263D 的具體分配）、座標軸轉換公式（如 Z-up to Y-up）等硬核技術細節。
    *   **保留歷史**：更新文檔時盡量以新增內容為主，除非原內容錯誤，否則不要複寫或縮短原有的紀錄。
3.  **進度對齊**：
    *   每三到五輪對話必須重讀 `docs/` 下的所有 `.md` 檔案，確保開發方向與規範一致。

## 4. 視覺化規範 (Visualization)
*   所有視覺化結果必須輸出至該數據目錄下的 `visualizations/` 資料夾。
*   渲染器應支援不同關節數（22/24/52j）的自動匹配。
*   在調試階段，應開啟關節點編號標註功能以排查索引問題。

## 5. 工作流程 (Workflow)
1. 探索與研究：先看 `external/` 中的現有邏輯。
2. 實作與紀錄：寫代碼的同時，同步更新 `RESEARCH_LOG.md`。
3. 驗證：必先用 `inspector.py` 驗證數據維度，再用視覺化確認幾何正確性。
4. 完工：更新 `PROGRESS.md` 狀態。
